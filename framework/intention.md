# Intention du Framework AIAD

*Document de cadrage pour la rédaction et l'évolution du Framework AIAD*

---

## 1. Vision

### 1.1 Objectif du Framework

Fournir une **méthodologie de référence** pour le développement produit à l'ère des agents IA, permettant aux équipes de :

- Maximiser la valeur livrée aux utilisateurs
- Exploiter pleinement le potentiel des agents IA
- Maintenir une qualité professionnelle sans sacrifier la vélocité
- Adopter une approche empirique et adaptative

### 1.2 Audience Cible

| Audience | Besoins Principaux |
|----------|-------------------|
| **Product Managers** | Cadre pour définir et mesurer la valeur, orchestrer la discovery |
| **Product Engineers** | Méthodologie d'orchestration des agents IA, standards de qualité |
| **Tech Leads** | Guidelines architecturales, gestion de la cohérence technique |
| **Agents Engineers** | Stratégies de configuration et d'optimisation de l'écosystème IA |
| **QA Engineers** | Stratégies de test adaptées au contexte IA |
| **Organisations** | Framework d'adoption et de transformation |

### 1.3 Positionnement

AIAD se positionne comme une **alternative moderne aux frameworks agiles traditionnels** (Scrum, Kanban) en intégrant nativement :

- L'orchestration d'agents IA comme pratique centrale
- L'empirisme radical plutôt que la planification rigide
- La fluidité plutôt que les cérémonies prescrites
- L'excellence produit plutôt que la conformité processus

## 2. Problèmes Adressés

### 2.1 Limites des Frameworks Actuels

| Problème | Impact |
|----------|--------|
| Rigidité des Sprints | Inadapté à la vélocité des agents IA |
| Cérémonies excessives | Temps perdu, frustration équipe |
| Focus sur l'output | Perte de vue de la valeur réelle (outcomes) |
| Rôles figés | Manque d'adaptabilité contextuelle |
| Absence de guidance IA | Exploitation sous-optimale des agents |

### 2.2 Opportunités Manquées

- Potentiel des agents IA sous-exploité (80%+ du code pourrait être généré)
- Feedback utilisateur trop tardif dans le cycle
- Dette de contexte (agents mal configurés, documentation obsolète)
- Silos entre discovery et delivery

## 3. Principes Directeurs de Rédaction

### 3.1 Ton et Style

- **Pragmatique** : Des recommandations actionnables, pas de la théorie abstraite
- **Direct** : Aller à l'essentiel, éviter la verbosité
- **Opinionated** : Prendre position, pas de "ça dépend" systématique
- **Inclusif** : Accessible aux débutants, utile aux experts

### 3.2 Structure

Chaque section du framework doit :

1. **Commencer par le "Pourquoi"** : Justifier l'existence de la section
2. **Définir le "Quoi"** : Concepts clairs et définis
3. **Guider le "Comment"** : Étapes actionnables
4. **Illustrer avec des exemples** : Cas concrets, templates
5. **Prévenir les erreurs** : Anti-patterns explicites

### 3.3 Niveau de Détail

| Section | Niveau | Justification |
|---------|--------|---------------|
| Principes et Vision | Conceptuel | Inspiration et alignement |
| Responsabilités | Pratique | Clarté sur les attentes |
| Artefacts | Détaillé | Templates utilisables |
| Boucles Itératives | Procédural | Étapes à suivre |
| Métriques | Quantifié | Cibles mesurables |
| Annexes | Exhaustif | Référence complète |

## 4. Critères de Succès du Framework

### 4.1 Adoptabilité

- **Compréhension rapide** : Framework compris en 30 minutes de lecture
- **Démarrage facilité** : Équipe opérationnelle en 1 semaine
- **Maîtrise progressive** : Expertise acquise en 3 mois

### 4.2 Efficacité

- **Amélioration mesurable** : gain de productivité en 3 mois
- **Qualité maintenue** : Bugs en production en baisse
- **Satisfaction équipe** : en hausse après adoption

### 4.3 Durabilité

- **Évolutif** : Framework adaptable aux évolutions des agents IA
- **Modulaire** : Adoption partielle possible
- **Auto-améliorant** : Mécanismes de feedback intégrés

---

## 5. Périmètre

### 5.1 In Scope

- Philosophie et principes fondateurs
- Définition des responsabilités (pas des rôles rigides)
- Artefacts essentiels avec templates
- Boucles itératives continues
- Synchronisations intentionnelles
- Métriques et amélioration continue
- Écosystème d'agents IA
- Guides d'implémentation (annexes)

### 5.2 Out of Scope

- Formation technique aux outils spécifiques
- Gestion RH et recrutement
- Aspects contractuels et légaux
- Détails de configuration de chaque agent IA (trop volatile)
- Comparaison détaillée avec autres frameworks

## 6. Versions et Évolution

### 6.1 Historique

| Version | Focus Principal |
|---------|----------------|
| v1.0 | Fondations du framework |
| v1.1 | Intégration des agents IA |
| v1.2 | Rupture avec Scrum, empirisme radical |
| v1.3 | Simplification, pragmatisme maximal |

### 6.2 Principes d'Évolution

1. **Rétrocompatibilité conceptuelle** : Les principes fondateurs restent stables
2. **Pragmatisme** : Changements basés sur le feedback terrain
3. **Minimalisme** : Ajouter uniquement si nécessaire, retirer si obsolète
4. **Transparence** : Changelog clair entre versions

### 6.3 Roadmap Indicative

- **v1.4** : Enrichissement des annexes, cas d'usage sectoriels
- **v2.0** : Évolution majeure basée sur 12+ mois de feedback

## 7. Contraintes et Risques

### 7.1 Contraintes

| Contrainte | Impact sur la Rédaction |
|------------|------------------------|
| Évolution rapide des LLMs | Garder les recommandations génériques |
| Diversité des contextes | Proposer des variantes selon taille d'équipe |
| Langue française | Terminologie cohérente, anglicismes maîtrisés |
| Format web | Sections autonomes, navigation facile |

### 7.2 Risques de Rédaction

| Risque | Mitigation |
|--------|------------|
| Trop théorique | Exemples concrets systématiques |
| Trop prescriptif | Principes > règles rigides |
| Obsolescence rapide | Abstraction des outils spécifiques |
| Incohérence interne | Relecture croisée, glossaire |

---

## 8. Glossaire Essentiel

| Terme | Définition |
|-------|------------|
| **Outcome** | Résultat mesurable de valeur pour les stakeholders |
| **Output** | Livrable produit (code, fonctionnalité) |
| **Product Engineer** | Développeur orchestrant des agents IA |
| **Agents Engineer** | Responsable de l'écosystème d'agents IA |
| **AGENT-GUIDE** | Document de contexte pour les agents IA |
| **SPEC** | Spécification technique détaillée |
| **DoOD** | Definition of Output Done |
| **DoOuD** | Definition of Outcome Done |
| **Boucle itérative** | Cycle PLANIFIER → IMPLÉMENTER → VALIDER → INTÉGRER |
| **Synchronisation** | Réunion intentionnelle (remplace "cérémonie") |

## 9. Checklist de Validation

Avant publication d'une section, vérifier :

- [ ] **Cohérence** : Alignement avec les principes AIAD
- [ ] **Clarté** : Compréhensible sans contexte supplémentaire
- [ ] **Actionnabilité** : Le lecteur sait quoi faire concrètement
- [ ] **Exemples** : Au moins un exemple concret par concept
- [ ] **Anti-patterns** : Erreurs courantes documentées
- [ ] **Liens** : Références croisées vers sections liées
- [ ] **Format** : Respect des conventions de style
